# 提示词编写原则

> 每次动笔前过一遍，写完后对照检查。

---

## 动笔前：先问自己三个问题

```
1. 模型扮演什么角色？（Who）
2. 它要完成什么任务？（What）
3. 边界情况怎么处理？（When things go wrong）
```

三个问题都想清楚，再开始写。

---

## 写的时候：六条原则

### 原则一：明确 > 隐含
不要依赖模型"理解你的意思"，把期望写出来。

| ❌ 模糊 | ✅ 明确 |
|--------|--------|
| 返回JSON格式 | gender 只能是 "男" 或 "女"，无法判断填 null |
| 用中文回答 | 始终用中文，代码注释也用中文 |
| 简洁一点 | 回答不超过200字，只给结论，不解释过程 |

---

### 原则二：约束"不能做"比描述"应该做"更有效
禁止语句比期望语句对模型的约束力更强。

| ❌ 弱约束 | ✅ 强约束 |
|---------|---------|
| 如果不知道请说明 | 禁止推测，信息不足时只说"无法确定" |
| 尽量准确 | 不得编造文档中未出现的内容 |

---

### 原则三：先写 Fallback，再写 Happy Path
大多数人只写正常流程，忘了边界情况。
但幻觉和错误几乎都发生在边界上。

**写完提示词，问自己：**
- 输入缺失/模糊时，模型会怎么做？
- 找不到答案时，它会乱编吗？
- 有没有明确告诉它遇到异常该怎么办？

---

### 原则四：结构化指令 > 大段散文
模型对有编号的列表遵循率更高，散文描述容易被忽略。

```
❌ 请你帮我提取文本中的人名、职业和城市，如果没有的话就填空，
   返回JSON，注意性别只能是男或女……

✅ 提取以下字段：
   1. name：人名，字符串
   2. occupation：职业，字符串，无法判断填 null
   3. city：城市，无法判断填 null
   4. gender：只能是 "男" 或 "女"，无法判断填 null
```

---

### 原则五：上下文越少，提示词要越详细
上下文信息量与提示词精确度要求成反比：

```
有代码 / 有文件 / 有示例  →  提示词可以简短
空对话 / 全新任务          →  提示词必须详细
```

判断方法：把提示词给一个从没见过这个项目的人看，他能准确执行吗？

---

### 原则六：给出示例是最快的"对齐"方式
当语言描述说不清楚期望时，直接给例子。

```
输入示例：张三，男，28岁，北京，Java开发
期望输出：{"name":"张三","gender":"男","age":28,"city":"北京","occupation":"Java开发"}

输入异常示例：一个工程师在某城市工作
期望输出：{"name":null,"gender":null,"age":null,"city":null,"occupation":"工程师"}
```

---

## 写完后：四个自检问题

```
□ 模型知道自己是"谁"吗？（角色）
□ 每个字段/要求都有明确定义吗？（精确）
□ 信息不足时模型知道该怎么做吗？（Fallback）
□ 把提示词给另一个人看，他的理解和你一样吗？（无歧义）
```

四个都打勾，才算写完。

---

## 优化时：迭代流程

```
发现一个坏案例
    ↓
分析根因（是角色不清？规则模糊？没有Fallback？）
    ↓
针对性修改提示词
    ↓
用同样的输入验证，确认修好了
    ↓
用旧的好案例回归，确认没有破坏原有效果
```

**关键：不要凭感觉改，要有具体的"这个输入之前坏了，现在好了"作为依据。**

---

## 一句话记忆法

> **提示词 = 把脑子里的期望，翻译成模型能执行的约束。**
> 翻译越精确，结果越可控。
